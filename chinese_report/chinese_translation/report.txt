本项目是一个使用 FastAPI、SQLite、Jinja2、HTML、CSS 和 JavaScript 开发的小型 Online Judge 系统。
该系统允许用户注册、登录、查看编程题目、提交 Python 解答并接收自动评测结果。

项目支持三种角色：
学生：
查看题目
提交解答
查看个人提交
查看安全的评测信息
教师：
创建题目
编辑题目
删除题目
查看提交
查看完整评测日志
重新评测提交

管理员：
执行所有教师操作
管理用户
更改角色
禁用或启用账户
查看审计日志
创建和恢复备份

认证和 Cookie

密码以哈希形式保存
注册时，密码使用 bcrypt 进行哈希。
登录时，输入的密码会与已保存的哈希值进行检查。
成功登录后，签名 session cookie 中只保存 user_id。


request.session["user_id"] = user["id"]
Cookie 不保存密码、密码哈希或完整用户记录。
对于每个受保护的请求，后端都会从 SQLite 加载当前用户数据。
这意味着角色更改和账户禁用会立即生效。
退出登录时，应用程序执行：
request.session.clear()
last_seen_at 的值会被保留。它不会被删除，因为它表示最后一次活动的真实时间。

基于角色的授权

前端会隐藏不可用的按钮，但后端仍会检查每一个受保护的操作。
学生不能创建、编辑或删除题目。
教师可以管理题目和评测结果。
管理员可以管理用户、备份和审计日志。
这可以防止用户通过手动发送请求来绕过权限。

教师和管理员可以创建编程题目。


题目标识符是包含字母、数字、下划线或连字符的字符串。一个题目包含一个公开样例和至少一个测试用例。
测试用例分数之和必须恰好为 100。隐藏测试只返回给教师和管理员。
题目删除使用软删除，因此历史提交仍然有效。

每个题目包含：
标题
描述
输入格式
输出格式
约束
样例
测试用例
时间限制
内存限制

系统检查：
测试用例 id 是唯一的
至少有一个测试
总分等于 100

删除的题目会从普通列表中消失，但旧提交和日志仍然可用。

提交和评测流程

当学生提交代码时：

1. 创建一条状态为 pending 的提交记录。
2. 后台任务启动评测器。
3. 状态更改为 running。
4. 源代码保存到临时 main.py 文件中。
5. 程序在单独的子进程中运行。
6. 输入通过 stdin 传递。
7. 捕获 stdout 和 stderr。
8. 将输出与预期输出进行比较。
9. 为每个测试创建日志。
10. 保存最终结果和分数。
11. 删除临时文件。

评测器不使用 eval() 或 exec()
学生代码在单独的 Python 进程中执行
系统只为通过的测试分配分数

每个测试用例保存结果、分数、时间、退出代码、输入、预期输出、stdout、stderr 和消息。长字段会截断到 4000 个字符。
学生不能接收隐藏输入、隐藏预期输出或隐藏 stdout。可见错误中的绝对路径会被移除。
管理操作保存在 `audit_logs` 中。


评测器在比较之前会标准化换行符。
Windows CRLF 换行符会转换为 LF。
会忽略行尾空格和末尾多余的空行。
会保留有意义的行首空格和内部空格。

时间限制适用于整个提交。
它不会为每个测试重新设置。
评测器计算：
剩余时间 = 总限制 已使用时间

例如：
总限制：2.0 秒
第一个测试：0.7 秒
剩余时间：1.3 秒

下一个测试只获得剩余时间。

如果没有剩余时间，结果为 TLE。

学生不能看到：
隐藏测试输入
隐藏预期输出
完整的隐藏 stdout
绝对服务器路径
教师和管理员可以查看完整日志。
错误信息在显示给学生之前会被清理。
评测输出在保存之前会被截断。


所有时间戳都使用 ISO 8601 格式以 UTC 保存。
示例：
2026-07-22T08:35:20Z


当已认证用户发送请求时，last_seen_at 字段会被更新。
在线状态根据最近活动计算。
项目不会永久保存 is_online。
退出登录只清除 session cookie，并保留 last_seen_at。
